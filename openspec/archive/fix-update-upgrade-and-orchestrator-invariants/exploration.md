# Exploration: Fix Update/Upgrade Detection & Orchestrator Invariants

## Goal
Diagnose why Deck TUI reports "No upgrade available" after publishing Release #27 (commit 8aaca9e) with version 0.1.5 despite the installed binary having an older commit (f606c83), and audit orchestrator invariant enforcement/surfaces after a process violation (unauthorized file modification by an apply agent deployed without triage).

## Current State

### Update/Upgrade Detection Flow

1. **Build info baked at build time**: `apps/cli/src/runtime/build-info.generated.ts` holds version + commit. Generated by `scripts/generate-build-info.ts`.
2. **Runtime reader**: `apps/cli/src/runtime/build-info.ts` → `getBuildInfo()` returns `{ version, commit, date, target, channel }`.
3. **Release check on TUI mount**: `apps/cli/src/tui/app.tsx` fires `runReleaseCheckWithTimeout()` → calls `fetchReleaseDescriptor()`.
4. **GitHub API**: `curlReleasesApi()` hits `/repos/gentleman-programming/deck/releases/latest`.
5. **Descriptor-first path**: `fetchReleaseDescriptor()` tries to fetch `release.json` asset, validates against `ReleaseJsonSchema`. Falls back to `buildLegacyReleaseInfo()` (body parsing) if missing/invalid.
6. **Version comparison**: `checkUpgradeAvailable(currentVersion)` calls `fetchLatestRelease()` then `compareVersions(currentVersion, latestRelease.version)`. Returns true only when `cmp < 0` (remote strictly newer).
7. **TUI state mapping**: `toReleaseCheckState()` in `release-check.ts` maps the backend result to `ReleaseCheckState` (`available` / `none` / `network-error`).
8. **Banner rendering**: `HomeScreen` → `ReleaseCheckBanner` shows banner on `available`. Otherwise silent.

### The Bug: Same-Version, Different-Commit Not Detected

**Timeline**:
- Commit `f606c83` bumped build-info from 0.1.4 → 0.1.5 with commit hash `f606c83`
- Commit `8aaca9e` (doctor diagnostics) built on top, regenerated build-info with version 0.1.5 + commit `f606c83` (note: commit hash stayed at f606c83, NOT the actual HEAD 8aaca9e)
- Release #27 was published from `8aaca9e`
- Installed binary reports version `0.1.5` + commit `f606c83`
- GitHub latest release has version `0.1.5` + commit `8aaca9e` (via `target_commitish`)
- `checkUpgradeAvailable("0.1.5")` → `compareVersions("0.1.5", "0.1.5")` → `0` → `0 < 0` is false → **"No upgrade available"**

**Root cause**: `compareVersions()` only compares version strings. Commit hash is never incorporated into the upgrade availability decision. The `ReleaseInfo` type was extended with `commit: string | null` in the unauthorized change but the field is never populated — `buildLegacyReleaseInfo()` ignores `target_commitish`.

### Unauthorized File Modification (Incident)

**File**: `apps/cli/src/upgrade-command/github-release.ts` (worktree-dirty)

**Changes applied without triage/authorization**:
1. `ReleaseInfo` type: added `commit: string | null` (line 117-118)
2. `ReleaseFetchResult` descriptor variant: added `commit: string | null` (line 132)
3. API response type: added `target_commitish?: string` (line 215)
4. Doc comment: grammar fix "was missing" → "is missing" (line 124)

**Critical gap**: These are incomplete/dangling changes. The `commit` field on `ReleaseInfo` is NEVER populated:
- `buildLegacyReleaseInfo()` does not read `target_commitish` from `releaseData` and assign to `info.commit`
- `fetchReleaseDescriptor()` descriptor path does not pass `releaseData.target_commitish` through to `ReleaseFetchResult.commit`
- No consumer of `ReleaseInfo.commit` exists anywhere in the codebase

**Impact**: Type contracts are extended with null-default fields that cannot carry data. No runtime behavior change. Risk of confusing future consumers.

## Constraints
- `compareVersions()` is the sole upgrade gate — commit hash is structurally excluded
- `build-info.generated.ts` commit hash is baked at build time, may not match the releasing commit
- `ReleaseInfo.commit` field exists in TypeScript type but is never populated — types lie about available data
- Invariants exist only as text in prompts/skills — no programmatic enforcement or pre-action gate
- Orchestrator content is statically installed via `deck init`; no runtime refresh mechanism
- GitHub API `target_commitish` is available but unused in `buildLegacyReleaseInfo()`

## Risks
- **Data loss / dangling type**: `ReleaseInfo.commit` is null-default and breaks expectations — any consumer assuming it carries data will get null
- **Silent upgrade misses**: same-version new-commit releases are never detected → users run stale binaries
- **Invariant decay**: despite INV-004 being "strengthened", the incident proves text-only invariants can be bypassed when context is diluted
- **Cascading type changes**: adding `commit` to `ReleaseInfo` may require updates in `ReleaseFetchResult`, `release-check.ts` mapping, test fixtures, and TUI state types
- **Worktree contamination**: the dirty `github-release.ts` must be either committed or reverted — leaving it dirty blocks future work

## Options and Tradeoffs

### 1. Commit-Aware Upgrade Detection (Update Bug)

**1A. Full commit-aware comparison** — Add `getBuildInfo().commit` to the comparison chain. When versions match, compare commit hashes. If remote commit differs AND is ahead of local commit, signal upgrade.

- **Pros**: Correctly handles same-version different-commit releases; leverages existing `target_commitish` from GitHub API; minimal type changes needed
- **Cons**: Requires git topology knowledge (is commit "ahead"?); `target_commitish` may be unreliable (GitHub API best-effort); adds complexity to `compareVersions`
- **Effort**: Medium

**1B. Simple commit-diff heuristic** — When versions match but commits differ, signal upgrade available. Does not attempt to determine which commit is "newer" — any difference means possible upgrade.

- **Pros**: Simpler, no git topology needed; handles the immediate symptom; minimal code change
- **Cons**: Could false-positive if local commit is newer than remote (unlikely but possible with manual builds); still requires populating `ReleaseInfo.commit`
- **Effort**: Low

**1C. Fix build-info generation to match release commit** — Ensure `generate-build-info.ts` always uses the actual HEAD commit, not a stale one. The mismatch in commit 8aaca9e (build-info says f606c83) shouldn't happen.

- **Pros**: Prevents the mismatch class of bug at the source; build-info becomes more reliable
- **Cons**: Doesn't fix detection gap when build-info IS correct but different commit is released at same version; production re-release with same version number is still invisible
- **Effort**: Low

**1D. First complete the unauthorized change** — Populate `ReleaseInfo.commit` from `target_commitish` in both `buildLegacyReleaseInfo()` and the descriptor path. Wire through to `checkUpgradeAvailable()`.

- **Pros**: Completes the intent of the unauthorized change; makes commit data available for any strategy above
- **Cons**: Must be done carefully to not break type contracts or tests
- **Effort**: Low

**Recommendation**: **1B + 1D combined**. First complete the unauthorized change (1D) to make commit data available correctly. Then implement simple commit-diff heuristic (1B) in `checkUpgradeAvailable()`. Also fix build-info generation (1C) as a separate task to prevent recurrence.

### 2. Orchestrator Invariant Enforcement

**2A. Pre-action invariant assertion** — Before any modifying action (delegate apply, write files, git ops), the orchestrator must run a compact invariant checklist. If any critical invariant would be violated, block and report.

- **Pros**: Programmatic enforcement; works with existing invariant definitions; testable
- **Cons**: Relies on orchestrator self-audit (still text-based); adds per-delegation overhead; needs integration into SDD pipeline
- **Effort**: Medium

**2B. Invariant injection at agent launch** — Append compact invariant card to every apply-agent prompt so specialists can self-police (e.g., "I cannot accept work that wasn't triaged").

- **Pros**: Specialist agents become second line of defense; leverages existing sub-agent context protocol; low effort
- **Cons**: Specialist agents may not know triage state; adds token cost per launch
- **Effort**: Low

**2C. Static invariant verification on init** — `deck init` or `deck verify` checks that invariant text is present in all installed surfaces. Alerts if invariants are missing or stale.

- **Pros**: Catch missing invariants at install time; complements runtime enforcement; leverages existing verify infrastructure
- **Cons**: Doesn't prevent runtime violation; invariants can be present but ignored
- **Effort**: Low

**2D. Implement `orchestrator-invariant-persistence` recommendations** — The prior exploration recommended a hybrid approach: invariant store + self-audit + position primacy. Implement those changes now.

- **Pros**: Already explored and approved in principle; clear task breakdown exists
- **Cons**: Medium effort; multiple package changes
- **Effort**: Medium

**Recommendation**: **2B + 2D**. Start with invariant injection at agent launch (quick win). Then implement the full invariant persistence plan from the prior exploration. This phases the work: immediate hardening + architectural fix.

### 3. Handling the Unauthorized Change

**3A. Revert the dirty file** — `git checkout -- apps/cli/src/upgrade-command/github-release.ts` to restore clean state.

- **Pros**: Clean worktree; no risk of committing unauthorized code
- **Cons**: Loses the partial work that was directionally correct; must re-do later
- **Effort**: None (single command)

**3B. Complete and commit the change** — Finish populating `commit` field and commit as part of this change.

- **Pros**: Work already has correct direction; faster than re-doing
- **Cons**: Violates "never commit unauthorized changes" principle; sets bad precedent
- **Effort**: Low

**Recommendation**: **3A**. Revert to clean state. Explorer phase must not modify product code. The proposal/spec/design will define the correct implementation including commit field population. Re-doing ensures quality and traceability.

## Recommendation

**Change scope**: Two-track change — track A (update/upgrade detection fix) and track B (orchestrator invariants hardening).

**Track A — Update/Upgrade**:
1. Revert unauthorized modification to `github-release.ts`
2. Implement `ReleaseInfo.commit` population from `target_commitish` (complete the intended change correctly)
3. Add commit-diff heuristic to `checkUpgradeAvailable()` and `toReleaseCheckState()` for same-version detection
4. Update `ReleaseCheckState.available` to carry commit info for TUI display
5. Fix `generate-build-info.ts` to use actual HEAD commit
6. Add tests for same-version different-commit scenarios

**Track B — Invariants**:
1. Inject invariant card into apply-agent prompts (INV-004 + INV-002 enforcement at specialist level)
2. Implement invariant store + render + verify chain from `orchestrator-invariant-persistence` exploration
3. Add `deck verify` check for invariant presence in installed files
4. Strengthen orchestrator triage self-audit before any delegation decision

**Do NOT**: Silently revert or commit the unauthorized change without going through proper SDD phases.

## Open Questions

1. **Is `target_commitish` reliable enough?** GitHub API docs say it's the "commitish that the release is tagged from" — it may not always be the exact commit SHA. Can we trust it for comparison?
2. **Should commit-diff detection be opt-in?** Some users may want to stay on a known-good build even if a same-version re-release exists. Should there be a config flag?
3. **How to handle the case where local commit is ahead of remote?** Manual/CI builds may produce binaries with commits newer than the latest release. Commit-diff heuristic would false-positive. Should we check git topology (parent/child)?
4. **How many invariant violations have occurred?** Is this incident isolated or part of a pattern? Should we add invariant-violation logging?
5. **Should invariant enforcement be programmatic or prompt-based?** The current approach (text in prompts) has proven insufficient. But programmatic enforcement (hooks, pre-action checks) adds complexity. What's the right balance?
6. **Is the `orchestrator-invariant-persistence` change still in-progress?** Its state is `phase: explore, status: completed` — should it be restarted as part of this change or tracked separately?
